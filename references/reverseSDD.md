## 倒推 SDD（从代码反推 PRD 和 Spec）

目的有两个：① 倒推规格的过程逼你回答"当初为什么这么设计"，是吃透项目的最佳方式；② 练"接手一个没有文档的陌生项目"的真实能力——这正是进公司后的日常。

> 重要区分：反推出来的 v1 PRD/Spec 是**逆向重建的学习产物**，不是你原创设计的产品。面试时大方讲"我反推规格来吃透架构"反而加分（证明你会做需求分析）。你后来自己加功能写的 v2 文档才是货真价实的原创。

---

### 第一步：倒推 PRD（做什么，不含技术）

PRD 回答"做什么、给谁、解决什么问题"。现成线索：README 的业务流程 + 系统截图 + 路由列表。

**PRD 模板**
```
# {项目名} 产品需求文档（PRD · 反推重建版）

## 1. 背景与目标
- 要解决什么问题
- 目标用户是谁

## 2. 用户角色
| 角色 | 是谁 | 主要诉求 |
| 求职者 | C 端用户 | 想要针对岗位的模拟面试和评分反馈 |
| 管理员 | B 端运营 | 维护题库、岗位模板、知识库 |

## 3. 核心用户故事
- 作为求职者，我上传简历后，希望系统解析并给出针对目标岗位的模拟面试和评分报告
- 作为求职者，我希望有 Agent 帮我做岗位匹配，让面试更有针对性
- 作为管理员，我希望能导入题目并向量化，供面试出题使用

## 4. 功能清单
| 模块 | 功能点 | 优先级 |

## 5. 非功能需求
- 评分要稳定、可解释（不能凭感觉）
- AI 输出要流式，减少等待焦虑
- 题库不足时流程不能中断（要兜底）

## 6. 不做什么（边界）
- 明确划出不在范围内的东西
```

---

### 第二步：倒推 Spec（怎么做，最有价值）

Spec 回答"怎么做"。证据全在代码里，直接读：

| Spec 内容 | 从哪读 |
|-----------|--------|
| 数据模型 | `app/models/` 的表定义 + `migrations/versions/` 的演进 |
| API 契约 | `app/api/` 的路由 + `app/schemas/` 的请求/响应模型 |
| 模块边界 | `app/services/` 的目录结构（client/backoffice/common 分法） |
| 关键技术决策 | 读代码里的兜底、单例、温度设置、召回策略 |

**Spec 模板**
```
# {项目名} 技术规格文档（Spec · 反推重建版）

## 1. 系统架构
{Mermaid 架构图：前端/后端/DB/缓存/任务队列/外部 AI}

## 2. 数据模型
| 表名 | 关键字段 | 说明 | 是否含向量 |
{逐表列出，标注外键关系}

## 3. API 契约
| 接口 | 方法 | 入参(schema) | 出参(schema) | 说明 |

## 4. 核心模块设计
### 4.1 岗位匹配 Agent
- 工具清单 + 每个工具的职责
- 编排方式（tool-calling + executor 配置）
- 关键参数（max_iterations、温度、错误透传）

### 4.2 RAG 出题
- 链路五段：向量化→存储→召回→选题→兜底
- 召回策略 + 三层兜底

### 4.3 AI 评分
- 评分 prompt 怎么构造（注入 reference_answer / key_points）
- 流式返回

## 5. 关键技术决策与理由（面试金矿）
| 决策 | 选了什么 | 为什么 | 备选方案 |
| 向量存储 | pgvector | 复用 PG，省一个独立组件 | Faiss/独立向量库 |
| 岗位匹配 | Agent tool-calling | 可解释、可调试、分步稳定 | 一个大 Prompt |
| 评分 | 注入采分点 | 降低随机性、可解释 | 让模型自由打分 |
```

### 第三步：模拟"用 Spec 喂给 Claude Code 分模块开发"

不需要重写整个项目。挑 1-2 个模块，体会"规格驱动开发"：

1. 挑一个模块（如"简历解析"或"RAG 出题"）
2. 把该模块的细粒度 spec 单独写出来（输入、输出、步骤、兜底）
3. 用 Claude Code 让它根据 spec **从零实现一遍**
4. 和原作者的实现**逐点对比**——差异处就是你的学习点和面试谈资

这个对比过程，就是你简历上"我理解并能复现这套架构"的底气。

---

### 目录与版本管理建议
- `specs/` 放反推的 PRD + Spec（v1）和你自己加功能的需求/规格（v2）
- `docs/` 放学习笔记（面试要点、RAG 原理、Agent 规划书）
- v1 文档走一个分支（如 `docs/reverse-engineered-specs`）单独提交
- README 里若引用了这些文档，记得更新链接路径
