## 倒推 SDD（从代码反推 PRD 和 Spec）

目的有两个：① 倒推规格的过程逼你回答"当初为什么这么设计"，是吃透项目的最佳方式；② 练"接手一个没有文档的陌生项目"的真实能力——这正是进公司后的日常。

> 重要区分：反推出来的 v1 PRD/Spec 是**逆向重建的学习产物**，不是你原创设计的产品。面试时大方讲"我反推规格来吃透架构"反而加分（证明你会做需求分析）。你后来自己加功能写的 v2 文档才是货真价实的原创。

⭐ **两份文档的验收标准（一开始就按这个标准写，不要先写个浅的再返工）**：
- **PRD** = 模拟"这个项目立项时产品经理该写出的文档"——语气是**规划将来**，不是描述现状
- **Spec** = **复现级**——他人不看源码、只照此文档能重建整个系统

---

### 第零步：补齐事实缺口（写之前必做，防编造）

反推文档最大的失败模式是"看起来像真的"：目录树是想象的、版本号写"最新"、接口契约靠猜。动笔前先把这些事实**从文件里读出来**：

| 缺口 | 去哪读 |
|------|--------|
| 目录树 | 实际扫一遍项目目录（到二级/三级） |
| 依赖版本 | `requirements.txt` / `pyproject.toml` 逐条抄 |
| 接口签名与前缀 | 路由注册文件 + `api/` 下全部路由定义 |
| 请求/响应契约 | `schemas/` 的 Pydantic 模型 + 统一响应包络 |
| 表结构与演进 | `models/` + `migrations/versions/` 按时间排序读一遍 |
| 配置默认值 | `config.py` + `.env.example` + `docker-compose*.yml` |

读的过程中发现的**原作者瑕疵**（死代码、配置默认值与实际部署不一致、重复定义）不要回避——记下来，写进 Spec 的"发现的实现瑕疵"一节，这是"真反推"区别于"抄 README"的最硬证据，也是现成的面试素材和二开选题。

---

### 第一步：倒推 PRD（做什么，不含技术）

PRD 回答"做什么、给谁、解决什么问题"。现成线索：README 业务流程 + 系统截图 + 路由列表 + **migrations 的表演进（反推版本史的金矿）**。

**PRD 模板（模拟真实需求文档）**
```
# {项目名} 产品需求文档（PRD · 反推重建版，文档性质要标注）

## 1. 版本演进史（依据 migrations 反推：哪个版本加了哪张表 = 加了什么能力）
| 版本 | 新增能力 | 对应表变更 |
| v0.9 | 账号体系 | users... |
| v1.0 | MVP：简历+面试+报告 | resumes/interviews... |

## 2. 背景与目标：要解决什么问题、目标用户是谁
## 3. 北极星指标 + 关键 KPI（如：面试完成率；辅以出题命中率/评分时延等）
## 4. 用户角色与旅程图（每个角色一条端到端旅程）
## 5. 功能需求（每条都带验收标准，用 Given/When/Then）
   例：Given 题库该岗位题目为 0，When 用户开始面试，Then 仍正常出题且题目标记 ai_fallback
## 6. 非功能需求（时延/稳定性/成本/可解释性）
## 7. 边界（明确不做什么）
## 8. 风险与依赖（如：单一 LLM 供应商、题库运营成本）
```

⚠️ 常见失败：写成"现有功能清单流水账"（那是说明书不是 PRD）；版本史凭空编（必须锚定迁移文件）。

### 第二步：倒推 Spec（怎么做，复现级）

Spec 的每一节都要回答"照着能不能做出来"。**分节生成，一次只写 1-2 节**——一口气生成全文，后半程（部署/验收清单）质量必然崩。

**Spec 模板（复现级 13 节）**
```
# {项目名} 技术规格文档（Spec · 复现级）

## 1. 系统架构 {Mermaid：前端/后端/DB/缓存/队列/外部 AI}
## 2. 技术选型与版本锁定 {逐条抄 requirements，禁止写"最新版"}
## 3. 目录结构规范 {与真实仓库一致的目录树，复现按此搭}
## 4. 配置规范 {config 全部关键项 + 默认值 + 哪些改错会炸}
## 5. 全局统一约定 {响应包络/鉴权方式/路由注册规则/横切纪律（如 LLM 调用必须走统一门面）}
## 6. 数据模型 {逐表字段级；标注向量列维度；最终以 migrations 为准}
## 7. API 契约全集 {分端列出：方法/路径/入参 schema/出参 schema/鉴权}
## 8. 核心模块详细设计 {每个模块伪代码级：分支条件、参数默认值、关键 SQL、
    prompt 的输出契约、失败兜底路径。这是全文最重的一节}
## 9. 前端规范 {端口/代理/关键页面/特殊协议（如 SSE 必须 fetch 不能 axios）}
## 10. 部署规范 {compose 服务表 + 启动序 + 已知坑}
## 11. 横切红线自检清单 {复现时逐条自检的硬约束}
## 12. 复现验收清单 {10 步左右，每步 = 可执行动作 + 可判定预期；
    要含反向用例（如"停用全部题库验证兜底生效"）}
## 13. 反推过程中发现的实现瑕疵 {第零步攒下的，诚实记录}
```

📎 颗粒度对照样例：`examples/specModuleSample.md`（一个真实项目的 §8 模块设计 + §12 验收清单节选，写之前先看它找感觉）。
📎 配套生成 prompt 见 `endToEndWorkflow.md` §3.4 / §3.5。

**自检**：写完拿第 12 节的验收清单模拟走一遍——如果某一步你自己都不知道预期结果，说明对应章节还没写到复现级。

### 第三步：模拟"用 Spec 喂给 Claude Code 分模块开发"

不需要重写整个项目。挑 1-2 个模块，体会"规格驱动开发"：

1. 挑一个模块（如"简历解析"或"RAG 出题"）
2. 把该模块的细粒度 spec 单独写出来（输入、输出、步骤、兜底）
3. 用 Claude Code 让它根据 spec **从零实现一遍**
4. 和原作者的实现**逐点对比**——差异处就是你的学习点和面试谈资

这个对比过程，就是你简历上"我理解并能复现这套架构"的底气。

---

### 目录与版本管理建议
- `specs/` 放反推的 PRD + Spec（v1）和你自己加功能的需求/规格（v2）
- `docs/` 放学习笔记（面试要点、RAG 原理、Agent 规划书）
- v1 文档走一个分支（如 `docs/reverse-engineered-specs`）单独提交
- README 里若引用了这些文档，记得更新链接路径
