## 学习路径模板（自学 + 求职导向，5 天速通版）

目标不是读懂每一行，而是**几天内能讲清几条主线 + 做出几个自己的改动 + 写出能过面试的简历**。
节奏可压缩可拉长，根据用户基础和时间灵活调整。

> 💡 **「以终为始」变体（实战验证过，推荐）**：不边读边问，而是 Day1 开始时把 Day1-5 的全部学习资料
> **一次性预先生成**（产物地图与每份资料的生成规范见 `endToEndWorkflow.md`），之后学习 =
> 读资料 → 做文末自测 → 打开源码对照 → 卡住再问。适合"不看代码先建认知"的学习者，也是弱模型下
> 质量最稳的打法（资料逐份生成、每份有必读清单和验收标准）。

### Day 0：先把项目跑起来（半天）
| 目标 | 具体行动 | 验证标准 |
|------|---------|---------|
| 能跑通 | 配 `.env`（DB/Redis/各 AI key）→ docker compose 起后端 → 跑迁移 `alembic upgrade head` → 前端 `npm run dev` | 浏览器能走通一遍完整业务流程 |

⚠️ 环境坑参考 `debugPlaybook.md`。AI 项目**必须先跑通**——不跑根本不知道数据长什么样。
🏆 里程碑：亲手走一遍 README 截图里的完整流程。

### Day 1：建全局地图（用模型当导师带一遍）
| 目标 | 具体行动 | 验证标准 |
|------|---------|---------|
| 看懂全景 | 用本 skill 画全景图 + 抽取核心主线 | 能用一句话说清每条主线的"输入→处理→输出" |

🏆 里程碑：不看代码能手绘出项目架构图。建议用更强的模型（如 Opus）做这次梳理，第一印象最值钱。

### Day 2：吃透两条命脉主线——Agent 和 RAG
| 目标 | 具体行动 | 验证标准 |
|------|---------|---------|
| 啃硬骨头 | 深读 Agent 编排 + 工具定义 + RAG 召回与兜底代码 | 能解释核心函数逻辑 + 能说清为什么这么设计 |

每读完一段就用 `interviewAmmo.md` 产出面试话术。这两块是简历命脉、面试必问。

### Day 3：倒推 SDD + 补文档
| 目标 | 具体行动 | 验证标准 |
|------|---------|---------|
| 反推规格 | 用 `reverseSDD.md` 生成 PRD + Spec，建 `specs/` 目录存放；补齐学习笔记到 `docs/` | 文档能让下一个新人看懂 |

🏆 里程碑：仓库里多出 `specs/`（反推的 PRD/Spec）和 `docs/`（面试要点、RAG 原理、Agent 规划书）。

### Day 4：二次开发——让它真正变成"你的"
| 目标 | 具体行动 | 验证标准 |
|------|---------|---------|
| 真改动 | 用 `secondaryDev.md` 选 1-2 个改动（补单测 / RAG rerank / 评分缓存），走 feature 分支 → PR → CI → 合并 | 改动能跑通、有测试、git 历史干净 |

🏆 里程碑：一条带 PR 和 CI 的真实提交历史——这是"真实项目经历"最硬的证据。

### Day 5：写简历 + 模拟面试
| 目标 | 具体行动 | 验证标准 |
|------|---------|---------|
| 输出成果 | 用 STAR 法则写项目经历 → 让模型扮演面试官连环追问（用你的 docs/specs 当题源） | 被追问"为什么这么设计"能答上 |

🏆 里程碑：一份能讲清 know-how 的项目经历 + 一轮扛得住深挖的模拟面试。

### 学习效果自检清单

**🗺️ 全局认知（Day 1）**
- [ ] 能用一句话说清项目做什么
- [ ] 不看代码能画出架构图
- [ ] 知道有哪些核心模块、各干什么
- [ ] 知道一个请求从进来到返回经过哪几步
- [ ] 知道用了什么技术栈

**🤖 AI 模块理解（Day 2）**
- [ ] 能讲清 Agent 的工具链和编排逻辑
- [ ] 能讲清 RAG 从召回到生成的五段链路
- [ ] 能说出召回不足时的兜底策略
- [ ] 能解释为什么用 tool-calling 而不是一个大 Prompt
- [ ] 能说清结构化输出失败怎么兜底

**✍️ 文档与二开能力（Day 3-5）**
- [ ] 能反推出像样的 PRD / Spec
- [ ] 知道加一个新功能要改哪些文件
- [ ] 能为自己的代码写单元测试
- [ ] 能评估改动对其他模块的影响
- [ ] 能用 STAR 把项目讲成面试故事

### 遇到困难时的求助策略

| 困难 | 建议行动 | 预计耗时 |
|------|---------|---------|
| 看不懂某个函数 | 先看注释和调用方，再问 AI | 5-10 min |
| 不理解设计意图 | 查 git log，或问 AI"为什么这样设计" | 10-15 min |
| 环境跑不起来 | 先看 README 和 `debugPlaybook.md`，再贴日志问 AI | 15-30 min |
| 改了不确定对不对 | 先跑测试，再让 AI 帮你 review | 10 min |
| 完全迷失方向 | 回到全局架构图，重新定位自己在哪 | 5 min |

### 💡 高效学习小贴士
1. 不要试图一次看懂所有代码——先看主干，细节用到再看
2. 多画图——画出来比脑子里想的清晰 10 倍
3. 带着问题看代码——"这个请求怎么处理的"比"看完这个文件"有效
4. 善用 git log——理解代码的演进历史
5. 模仿是最好的学习——找类似的已有实现照着改
6. 不怕问"蠢问题"——问比闷头看 3 小时高效
7. 写笔记——今天懂的，明天可能就忘了（这也正好变成你的 docs/）
