## 架构讲解检查清单（Python/AI 项目版）

讲解项目架构时，按此清单逐项确认，确保建立完整认知。不必每项都展开讲，但每项都要心里有数。

⭐ **产出架构文档时的两条原则（实战教训）**：
1. **独立成篇、自成体系**——不对标、不引用项目里已有的架构文档（哪怕存在），图和组织全部重新做；
   客观重叠的内容（如技术栈）不回避。学习资料的价值在于体系完整，"差异化补充"会让文档没法单独读。
2. **数字先核实**——本清单第 9/10/11 项涉及的配置默认值、迁移数、测试真假，写之前必须打开
   对应文件确认（config / migrations/versions/ / tests/），不凭印象填。

### 1. 项目定位（先回答"这是什么"）
- [ ] 一句话说清项目做什么、服务谁
- [ ] 标注核心业务场景（2-3 个）
- [ ] 解释"为什么需要这个项目"（业务价值）
- [ ] 判断工程成色：教程克隆 / 单 Prompt demo / 个人作品集 / 准生产 / 生产系统

### 2. 业务理解（先懂业务再看代码）
- [ ] 列出核心业务场景（用户视角）
- [ ] 用「用户故事」描述最常见的操作流程
- [ ] 解释关键业务术语（领域语言，如"采分点""候选人画像"）
- [ ] 标注"哪些是核心流程，哪些是辅助流程"
- [ ] 画出业务流程图（纯业务视角，不涉及代码）

### 3. 架构模式识别
- [ ] 判断项目类型：前后端分离 / 分层 / 单体 / 是否有异步任务
- [ ] 画出整体架构图（前端 / 后端 / DB / 缓存 / 任务队列 / 外部 AI 服务）
- [ ] 标注各部分通信方式（HTTP / SSE 流式 / 任务队列）
- [ ] 解释"为什么选这种架构"

### 4. 入口与启动
- [ ] 找到项目入口（如 `create_app()`、路由注册文件）
- [ ] 说明启动时做了哪些初始化（配置加载、DB 连接、中间件挂载、Celery 注册）
- [ ] 给出本地启动的最简步骤（docker compose / 迁移 / 前端 npm）
- [ ] 区分"改了要重启"还是"热加载"

### 5. 分层结构（这个项目的骨架）
- [ ] 识别分层模式（如 `api` → `services` → `models`，外加 `schemas` 做出入参）
- [ ] 用类比解释每层职责（api=前台接待，service=后厨，model=仓库，schema=验菜员）
- [ ] 画分层依赖图，标注调用方向
- [ ] 强调"上层调下层，下层不调上层"的规则
- [ ] 给一个"从 api 到 model"的完整示例，跟着走一遍
- [ ] 注意是否有 client / backoffice（用户端/管理端）双端隔离

### 6. 核心模块（按重要性排序）
- [ ] 识别 3-5 个核心模块（如 简历解析 / 岗位匹配 Agent / RAG 出题 / AI 评分 / 报告）
- [ ] 每个模块一句话职责边界
- [ ] 标注模块间依赖关系
- [ ] 区分"⭐必看"和"用到再看"
- [ ] 找到"AI 能力中枢"文件（统一封装所有 LLM 调用的那个）

### 7. AI 专项（Python/AI 项目特有，重点）
- [ ] Prompt：system/user 怎么分工、温度怎么设、怎么逼出结构化输出
- [ ] Agent：有几个工具、工具 docstring 怎么写、怎么编排、怎么防死循环
- [ ] RAG：灌数据→向量化→存储→召回→生成 五段链路，召回不足怎么兜底
- [ ] 流式输出：哪些接口用了 SSE，体验上为什么要流式
- [ ] 结构化输出兜底：模型不吐纯 JSON 时怎么救（这是高频面试点）

### 8. 数据流
- [ ] 说明数据从哪来（API / DB / 缓存 / 上传文件 / 外部 AI）
- [ ] 画一条典型请求的完整数据流转（如"上传简历→解析→落库"）
- [ ] 标注数据在每一跳的变化（入参→出参）
- [ ] 标注哪些数据持久化、哪些是临时的

### 9. 配置与环境
- [ ] 找到配置（`.env.example`、`config.py`、`docker-compose*.yml`）
- [ ] 解释关键配置项（DB / Redis / 各 AI 服务 key / 召回因子等）
- [ ] 区分本地开发配置 vs 线上配置
- [ ] 提醒哪些配置改错会出大问题（如向量维度、模型名）

### 10. 数据模型与迁移
- [ ] 找到 `models/` 下的表定义，列出核心表（用户/简历/题库/岗位模板/面试/报告）
- [ ] 看 `migrations/versions/` 有几个版本，表结构怎么演进的
- [ ] 标注哪些表存了向量字段（pgvector）
- [ ] 说明表与表之间的关系（外键/关联）

### 11. 工程化设施
- [ ] 有没有 CI（`.github/workflows/`），跑了哪些检查
- [ ] 有没有测试（`tests/`），是真测试还是占位模板 ⚠️
- [ ] 有没有多环境部署（dev/prod compose）
- [ ] 有没有日志/监控配置
- [ ] **诚实标注短板**：哪些声称的工程化是真的、哪些是空架子（面试官会问）

### 12. 扩展点（自学者/二开最关心）
- [ ] "如果要加一个新接口，改哪些文件"（给具体步骤）
- [ ] "如果要给 RAG 加一层 rerank，从哪入手"
- [ ] "如果要补单元测试，先测哪个函数最划算"
- [ ] **指出一个最简单的已有实现作为模板**，照着抄

### 13. 实操验证
- [ ] 给一个 30 分钟内能完成的"动手试试"小任务
- [ ] 完成后能验证对架构的理解
- [ ] 示例："找到简历上传接口的入口函数，追踪到它把 parsed_content 写进哪张表，画出这条调用链"
