## 二次开发选题（让 fork 项目真正变成"你的"）

fork 项目能不能上简历的分水岭，就是你有没有做**真实的、自己的改动**。
本文件给出"简历价值高 / 难度可控"的选题，以及标准的 git 工作流。

⭐ **规划与执行分离（实战验证的节奏）**：学习阶段先产出 `docs/04-二次开发规划.md`——
只列选题（背景/最小改动路径/验证标准/面试怎么讲），**不写实现代码**；等学习者明确说
"开始二开选题 X"再动手。一边学一边写代码，两头都做不深。
选题原则：优先**补短板型**（如补单测，顺便证明读懂了核心逻辑）和**小而硬型**（缓存/rerank/可观测性），
拒绝大而空（重构、换框架）。

---

### 推荐选题（按"简历价值 ÷ 难度"排序）

| 选题 | 改什么 | 难度 | 耗时 | 简历怎么讲 |
|------|--------|------|------|-----------|
| **补单元测试** | 把空壳 `tests/` 换成真测试，先测纯函数（如 `_extract_json` 的各种兜底分支） | ⭐ 低 | 半天 | "发现项目测试覆盖几乎为零，补充了核心解析/兜底逻辑的单元测试，接入 CI" — 直接补最大短板，面试加分 |
| **RAG 评分缓存** | 相同 query 的召回结果/评分加 Redis 缓存，命中直接返回 | ⭐⭐ 中 | 半天~1天 | "为高频重复的召回/评分加了缓存层，减少重复的 embedding 和 LLM 调用、降低成本和延迟" |
| **RAG rerank 重排** | 召回 Top-N 后，加一层重排（用 cross-encoder 或让 LLM 打分）再取 Top-K | ⭐⭐⭐ 中高 | 1~2天 | "在向量召回后增加重排序，提升题目与岗位的相关性，讲清 recall 和 precision 的权衡" |
| **可观测性** | 给 AI 调用加结构化日志/耗时统计/token 统计 | ⭐⭐ 中 | 半天 | "为 LLM 调用增加可观测性，能定位慢调用和成本热点" |
| **召回评估脚本** | 写个小脚本，用一批 query 评估召回命中率 | ⭐⭐ 中 | 半天 | "搭了简单的离线评估，量化 RAG 召回质量，为调参提供依据" |

**新手建议起步顺序**：先做"补单元测试"（最稳、补短板、练手），再做"评分缓存"或"rerank"（有技术含量、面试好讲）。

### 标准 git 工作流（这套历史本身就是你的"项目经历"证据）

哪怕一个人开发，也走完整流程，体现工程习惯：

```bash
# 1. 从 main 切 feature 分支
git checkout main
git pull
git checkout -b feature/add-unit-tests       # 命名：feature/功能

# 2. 开发 + 提交（提交信息写清楚做了什么）
git add .
git commit -m "test: add unit tests for _extract_json fallback branches"

# 3. 推到自己仓库
git push -u origin feature/add-unit-tests

# 4. 在 GitHub 开一个 PR（哪怕自己审自己，也体现 review 习惯）
#    等 CI 跑过，自己在 PR 里写一句 review 结论

# 5. 合并回 main
#    （GitHub 上点 Merge，或本地 merge 后 push）
```

**分支命名建议**
- `feature/add-unit-tests`
- `feature/rag-score-cache`
- `feature/rag-rerank`
- `docs/reverse-engineered-specs`（放反推的 PRD/Spec）

**版本管理建议**
- 每个功能一个分支、一个 PR，方便回退和讲述
- v1（反推文档）和 v2（自己加的功能）分开提交，历史清晰
- commit message 用规范前缀：`feat:` `fix:` `test:` `docs:` `refactor:`

### 二开时的安全提醒
- 改之前先 `git diff` 看清现状，先评估影响范围再动手
- AI key、密码这类敏感信息别提交进 git（检查 `.env` 在 `.gitignore` 里）
- 改完跑一遍完整业务流程，确认没弄坏主线
