# 以终为始工作流（自包含 SOP · 模型无关版）

> **这是什么**：一套把「吃透一个陌生 Python 项目 → 写进简历 → 扛住面试」全流程固化下来的操作手册。
> **怎么用**：① 在 Claude Code 里配合本 skill 自动执行；② 单独把本文（或某一节的 prompt）复制给**任何 AI**（Sonnet / DeepSeek / ChatGPT）也能跑；③ 完全没有 AI 时，照 §7 checklist 人肉执行。
> **核心思想（以终为始）**：不要边读代码边零散提问——先把**全部学习资料一次性生成**（Day1-5 每天一份主线资料 + 面试三件套），然后学习 = 读资料 → 做文末自测 → 打开源码对照 → 卡住再问。资料是航道，代码是验证。
> **本文经过一次完整实战验证**（一个 FastAPI + LangChain + pgvector 的 AI 面试项目，5 天走完全程），文中所有「示例」均来自该实战。

---

## §1 总流程

```
Day0 跑通 ──→ Day1 全景 ──→ Day2 核心专项 ──→ Day3 倒推SDD ──→ Day4 二开 ──→ Day5 面试输出
（部署+踩坑）  （架构+数据流）（最难最值钱的2-3条主线）（PRD+Spec）  （只规划后动手）（逐字稿+模拟面试）
```

| 阶段 | 进入条件 | 退出条件（没达到就顺延，"5 天"是任务量不是自然日） |
|------|---------|------------------------------------------------|
| Day0 | 拿到仓库 | 本地跑通完整业务流程；踩的坑写成复盘 |
| Day1 | 已跑通 | 不看资料能手绘架构图、讲清一条请求全路径 |
| Day2 | 有全景认知 | 核心主线的关键函数能逐段解释 + 说清设计取舍 |
| Day3 | 懂核心主线 | specs/ 两份文档达到"他人可照着复现"；亲手挑刺修订至少 1 处 |
| Day4 | 有规格认知 | 至少 1-2 个真改动合入 feature 分支，测试可跑 |
| Day5 | 有真改动 | 模拟面试被追问三层不崩；STAR 简历稿成型 |

**执行纪律**：资料生成阶段（通常 Day1 开始时一次性做完）**一次只生成一份文档**，生成前先完成该文档的「必读文件清单」（见 §3 各节）。禁止一条 prompt 要求生成全套——再强的模型也会在后半程开始编造。

## §2 标准产物地图（照搬到任何新项目）

```
{项目根}/
├── docs/
│   ├── 五天学习计划.md              ← 总导航（最后生成，索引其他所有资料）
│   ├── 00-部署实战与面试弹药.md      ← Day0 踩坑复盘 + STAR 素材
│   ├── 01-整体架构与数据流.md        ← Day1 主线（对照 architectureChecklist.md 13 项）
│   ├── 02-核心专项深潜.md            ← Day2 主线（AI 项目即 RAG/Agent/评分 等）
│   ├── 03-倒推SDD导读.md            ← Day3 主线（教你怎么用 specs/ 学习）
│   ├── 04-二次开发规划.md            ← Day4 选题清单（只规划，动手另起）
│   ├── 05-模拟面试问答实战.md        ← Day5 主线（口语逐字稿+追问链+避雷）
│   └── 基础补全/                    ← 基础知识字典（按学习者短板定制，随查随用）
│       ├── 01-技术框架概念补全.md
│       └── 02-代码架构与设计模式补全.md
├── specs/
│   ├── PRD.md                      ← 反推的产品需求（模拟真实需求文档）
│   └── Spec-技术规格.md             ← 反推的技术规格（复现级）
└── 面试要点.md                      ← 面试速查剪贴板（放根目录，考前 30 分钟扫）
```

**生成顺序**（依赖关系决定，不可乱）：
基础补全（定制给学习者）→ 01 全景 → 02 深潜 → PRD → Spec → 03 导读 → 04 二开规划 → 面试要点 → 05 问答实战 → 五天学习计划（收口）。

## §3 每份产物的生成规范

> 每节 5 要素：目的 / 生成前必读 / 可复制 prompt / 验收标准 / 常见失败。
> prompt 中 `{...}` 为占位符；示例值来自实战项目，换项目时替换。

### 3.0 docs/00-部署实战与面试弹药.md（Day0 产物）

- **目的**：部署踩坑是最硬的"真做过"证据，趁热复盘，直接变 STAR 素材。
- **必读**：docker-compose*.yml、README 部署章节、本次部署的报错记录/终端历史。
- **prompt**：
  ```
  我刚在 {环境，如 Ubuntu VM} 上部署完这个项目，过程中踩了这些坑：{逐条贴报错和解决过程}。
  帮我写 docs/00-部署实战与面试弹药.md：每个坑一节，按「现象 → 定位过程 → 根因 → 修复 → 这个坑教会我什么」组织；
  文末把最有含金量的 2-3 个坑整理成 STAR 故事。只写真实发生过的，不要补充"常见坑"凑数。
  ```
- **验收**：每个坑都有"定位过程"（不是只有结论）；STAR 故事的 R 有具体产出。
- **常见失败**：AI 往里塞没发生过的"常见坑"充数——prompt 里必须写"只写真实发生过的"。

### 3.1 docs/01-整体架构与数据流.md（Day1 主线）

- **目的**：一份**自成体系**的全景导读（不依赖读过任何其他文档），建立完整认知地图。
- **必读**（生成前 AI 必须先读，防编造）：入口文件（main/create_app）、路由注册、config、docker-compose、requirements/pyproject、models/ 目录、migrations/ 版本数、tests/ 是真是假。
- **prompt**：
  ```
  为这个项目生成 docs/01-整体架构与数据流.md，要求独立成篇、自成体系。
  严格对照 architectureChecklist.md 的 13 项逐项覆盖（项目定位/业务理解/架构模式/入口启动/分层/
  核心模块/技术专项/数据流/配置/数据模型/工程化/扩展点/实操验证）。
  重点画透：① 全景架构图 ② {数据生产端}→{数据消费端} 的数据流三通道 ③ {核心技术，如 AI/RAG/Agent} 的介入点。
  Mermaid 节点用中文、单图≤15 节点；标注 ⭐必看/⚠️易错/🎤面试点；
  「工程化」一节必须诚实标注短板（如 tests 是否占位）；文末给 30 分钟实操验证任务。
  所有端口、阈值、版本号、迁移数必须来自你刚读过的文件，禁止凭印象写。
  ```
- **验收**：13 项 checklist 各有对应章节；不读其他文档能看懂；含诚实短板一节。
- **常见失败**：把 README 复述一遍（没有自己组织的图和主线）；Mermaid 超载渲染不出；端口/版本号编造。

### 3.2 docs/基础补全/（按短板定制的字典）

- **目的**：学习者框架/领域知识生疏时，把项目涉及的概念**锚定到本项目真实代码**讲一遍——不是从零学教程。
- **必读**：requirements/pyproject（列出全部技术栈）、学习者自述的短板清单。
- **prompt**：
  ```
  我的基础情况：{如 Python 语法还行，但 FastAPI/异步/向量数据库/消息队列都没用过}。
  生成 docs/基础补全/01-技术框架概念补全.md：把本项目用到的核心技术概念按依赖顺序排列
  （先讲被依赖的，如 异步→框架→ORM→…），每个概念 = 大白话解释 + 本项目哪个文件怎么用它（给真实路径和函数名）
  + 面试可能怎么问。不要写成通用教程，每一条都必须落到本项目代码。
  ```
  02（代码架构与设计模式）同理：只讲**本项目实际出现**的分层与模式，每个模式给项目内真实锚点。
- **验收**：每个概念都有本项目文件路径锚点；顺序符合依赖关系。
- **常见失败**：写成脱离项目的百科教程；罗列项目里根本没用到的模式。

### 3.3 docs/02-核心专项深潜.md（Day2 主线）

- **目的**：把 2-3 条**最难、面试最值钱**的主线（AI 项目 = RAG/Agent/流式评分；其他项目 = 并发模型/核心算法/数据一致性等）啃到"逐段解释 + 说清取舍"。
- **必读**：每条主线的核心函数**全文**（如 RAG 的召回函数、生成分支函数；Agent 的工具定义与 Executor 组装；流式接口的生成器）。
- **prompt**：
  ```
  生成 docs/02-核心专项深潜.md，深潜这三条主线：{主线1/主线2/主线3}。
  每条主线：① 在全局中的位置（小图）② 核心代码走读——贴真实代码片段（从源文件复制，不要默写），
  逐段講清在做什么、为什么 ③ 关键设计取舍表（怎么做的/为什么/不这么做会怎样）④ 易踩坑
  ⑤ 面试弹药三件套（一句话讲法 + STAR + 追问预演）。
  文末出 10-15 道自测题（只列题不给答案，答案在正文里）。
  所有数字（阈值/重试次数/参数默认值）必须来自源码，写之前先读对应 config 与函数。
  ```
- **验收**：代码片段与源文件逐字一致；每条主线有取舍表；文末有自测题。
- **常见失败**：代码是"默写"的（与源码有出入，面试引用会翻车）；只讲"是什么"不讲"为什么"。

### 3.4 specs/PRD.md（反推产品需求）

- **目的**：模拟"这个项目立项时该有的 PRD"，练接手无文档项目的能力；版本史从 migrations 反推。
- **必读**：migrations/versions/ 全部迁移（按时间排）、路由清单、前端页面清单。
- **prompt**：
  ```
  从现有代码反推 specs/PRD.md，写成真实产品经理会写的样子：
  ① 版本演进史——依据 migrations 的表演进反推（哪个版本加了什么能力）② 北极星指标 + 关键 KPI
  ③ 用户角色与旅程图 ④ 功能需求（每条用 Given/When/Then 验收）⑤ 非功能需求 ⑥ 边界（明确不做什么）⑦ 风险。
  语气是"规划将来"而不是"描述现状"；标注文档性质：这是从代码逆向重建的学习产物。
  ```
- **验收**：版本史与迁移文件对得上；每条功能需求有验收标准；有"不做什么"边界。
- **常见失败**：写成功能清单流水账；版本史凭空编（必须锚定迁移文件）。

### 3.5 specs/Spec-技术规格.md（复现级，最高标准）

- **目的**：达到"**他人不看源码、只照此文档能重建系统**"的程度——这是检验是否真吃透的终极标尺。
- **必读**：requirements（锁版本）、目录树、config 全文、核心表模型、全部路由签名、核心模块函数、compose 文件。
- **prompt**（分节生成，一次一节，防止后半程质量崩塌）：
  ```
  反推 specs/Spec-技术规格.md，标准是"他人可照文档复现"。分节生成，本次先写第 {N} 节。
  全文结构：1 系统架构 2 技术选型与版本锁定 3 目录结构规范（与真实仓库一致）4 配置规范
  5 全局统一约定（响应包络/鉴权/路由注册/横切纪律）6 数据模型 7 API 契约全集
  8 核心模块详细设计（伪代码级，含关键 SQL/prompt 输出契约/参数默认值）9 前端规范 10 部署规范
  11 横切红线自检清单 12 复现验收清单（10 步可执行验证）13 反推中发现的实现瑕疵（诚实记录）。
  版本号抄 requirements，不要写"最新版"；发现原作者的死代码/不一致，记入第 13 节而不是回避。
  ```
- **验收**：拿第 12 节验收清单能实际执行；版本号与 requirements 一致；有"发现的瑕疵"一节（这一节反而是面试亮点：证明是真反推不是抄 README）。
- **常见失败**：一次生成全文导致 8-13 节质量骤降（必须分节）；API 契约漏鉴权/前缀；目录树是想象的。

### 3.6 docs/03-倒推SDD导读.md + docs/04-二次开发规划.md

- **03 目的**：教学习者**怎么用** specs/ 学习（复述→对照页面挑刺修订→遮住"为什么"自答→选做复现）。
- **04 目的**：只列二开选题不动手。**必读**：tests/ 现状、02 深潜里发现的短板。
- **04 prompt**：
  ```
  生成 docs/04-二次开发规划.md：给出 3-4 个能写进简历的二开选题，按「投入产出比」排序。
  每个选题：背景（现状缺什么）/ 最小改动路径（动哪几个文件、大致怎么改）/ 验证标准 / 面试怎么讲。
  选题原则：优先补短板型（如补单测——顺便证明读懂了核心逻辑）和小而硬型（如给检索加缓存/rerank），
  不要大而空（重构/换框架）。只规划，不写实现代码。
  ```
- **验收**：每个选题有最小改动路径和验证标准；没有"重构整个 XX"式大坑选题。

### 3.7 面试要点.md（根目录速查）+ docs/05-模拟面试问答实战.md（Day5 主线）

两份职责严格区分（见 §5），**先生成面试要点，再生成 05**（05 需要引用它做分工声明）。

- **面试要点 prompt**：
  ```
  生成根目录 面试要点.md（考前 30 分钟速查剪贴板）：
  ① 项目一句话版 + 简历三行版 + 诚实路径话术（"基于开源深入研究+二次开发"，绝不说"从零独立开发"）
  ② 亮点表：每个亮点 = 一句话讲法 + 被深挖时的支撑点（双栏）③ 2-3 个 STAR 故事
  ④ 高频追问清单（20 条左右，每条 = 简答一句 + 指路到哪份文档看细节）⑤ 反问环节问题 ⑥ 考前 30 分钟清单。
  ```
- **05 prompt**（这份文档的风格要求最特殊，逐字保留以下约束）：
  ```
  生成 docs/05-模拟面试问答实战.md。核心要求：避免官方生硬的标准答案，以真实做过这个项目的人的
  视角讲 know-how。每道题三件套：
  🎙️ 口语逐字稿——第一人称说话方式，不是文档腔；里面必须自然嵌入三类细节：具体数字（阈值/维度/参数）、
     踩过的坑（"我一开始…后来才发现…"）、取舍口头禅（"这么选是因为…代价是…什么时候该换成…"）
  🔍 追问链——面试官会顺着追的 2-3 层问题，每层给接法
  ⚠️ 避雷——这题最容易露馅的答法
  结构：开场 60 秒 → 按模块分区（每区 3-4 题）→ 压力面专区（"是你从零写的吗"必须有，答案是诚实路径）
  → 快问快答表 → 文末附"让 AI 当面试官"的自训 prompt。
  所有逐字稿里的数字先对照源码/前面文档核实。
  ```
- **验收**：逐字稿读出来像说话不像念文档；每题有追问链和避雷；压力面有"从零写的吗"；数字与源码一致。
- **常见失败**：答案写成百科式标准答案（一背就穿）；数字与 02/Spec 不一致（面试自相矛盾）。

### 3.8 docs/五天学习计划.md（总导航，最后收口）

- **目的**：索引全部资料 + 每日安排 + 总自检清单 + 卡住怎么办。所有资料生成完后最后写，才能索引齐。
- **验收**：资料地图与实际文件一一对应；每天有"验证标准在哪"一列。

## §4 事实核实纪律（弱模型最重要的护栏）

**规则：任何数字、阈值、分支条件、版本号、端口、表数/迁移数，写入文档前必须先读对应源文件。禁止"凭印象""按惯例"填。**

每个项目开工时先建一张「必核事实清单」，生成任何文档前对照：

| 类别 | 去哪核 | 实战示例（换项目时重填） |
|------|-------|------------------------|
| 配置默认值 | config.py / settings | min_score=0.7、RECALL_FACTOR=2、维度 1024 |
| 端口 | docker-compose*.yml | PG 5435、API 8006（config 默认 8001 是占位，以 compose 为准） |
| 依赖版本 | requirements/pyproject | fastapi 0.115.12、langchain 0.3.7 |
| 数据层 | migrations/versions/ 数一遍 | 9 个迁移、13 张表 |
| 分支条件 | 核心函数源码 | cnt>=total / 0<cnt<total / cnt==0 三分支 |
| 工程成色 | tests/ 打开看 | 是真测试还是占位模板（面试必问，必须诚实） |

**为什么这条对弱模型生死攸关**：强模型编造率低还常主动查证；弱模型会自信地填一个"合理值"（如 top_k=5、阈值 0.75），这些错数字进了面试材料，被面试官现场对出来就是灾难。把"先读文件"写死在流程里，模型强弱都能出对的数。

## §5 文档分工防重复策略

三份面试材料职责严格分层，生成时在各自开头声明分工，互相链接而不是重复内容：

| 文档 | 角色 | 什么时候用 |
|------|------|-----------|
| 面试要点.md | **速查剪贴板**：一句话讲法+简答 | 考前 30 分钟扫一遍 |
| docs/02-核心专项深潜.md | **技术底料**：代码级细节+取舍 | 平时精读；答不出回去翻 |
| docs/05-模拟面试问答实战.md | **张口说什么**：口语逐字稿 | 面试前一天对着练 |

同理：01（全景）与 Spec（规格）的区别 = 导读视角 vs 复现视角；发现两份文档在写同一块内容时，砍掉一边、换成链接。

## §6 弱模型 / 无 Claude Code 适配

**换弱模型时（Sonnet/DeepSeek 等）**，在 §3 各 prompt 基础上加三个动作：
1. **拆更细**：大文档（Spec、05）按节生成，一次一节；每节生成后人工扫一眼再要下一节。
2. **先喂样例**：生成前把 `examples/` 里对应的样例贴进对话（"照这个格式和颗粒度写"）——few-shot 对弱模型的质量提升远大于加长指令。
3. **生成后自查**：让模型对照该节「验收标准」自查一遍再交付（"逐条检查你刚才的输出是否满足：…"）。

**脱离 Claude Code 时（网页版 AI / 其他工具）**：模型读不到你的文件，你来当它的手——
1. 把本文对应节的 prompt 贴进对话；
2. 把「必读文件清单」里的文件内容手动贴上（config 全文、核心函数全文、requirements——通常 3-5 个文件就够一份文档）;
3. 数字核实由你执行：模型写完后，你拿源文件对一遍所有数字。

**skill 本身的维护策略**：用**当时能用到的最强模型**改编/升级本 skill（一次性投入），日常执行交给任何可用模型照 checklist + 样例跑。本 skill 就是这么来的（强模型从 Go 实习带教版改编 → 实战一轮 → 强模型沉淀本 SOP）。

## §7 移植到新项目的启动 checklist（10 步）

1. 把整个 `ai-project-tutor/` 文件夹拷入新项目 `.claude/skills/`（或确认全局已装）。
2. **先跑通项目**（Day0 铁律）：配 env → 起服务 → 浏览器走通一遍核心业务流程；踩坑随手记。
3. 生成 docs/00 部署复盘（§3.0）。
4. 让 AI 做一次工程成色判断（值不值得吃透、能不能写简历）——不值得就趁早换项目。
5. 建「必核事实清单」（§4 的表，逐项填上这个项目的真实值）。
6. 自述基础短板 → 生成 docs/基础补全/。
7. 说「**以终为始，按产物地图把学习资料全部生成**」→ AI 按 §2 顺序逐份生成（一次一份，每份先过必读清单）。
8. 逐日学习：读资料 → 文末自测 → 打开源码对照 → 卡住把原文/代码贴给 AI 追问。
9. Day4 动手二开（走 feature 分支），Day5 用 05 的自训 prompt 让 AI 当面试官车轮战。
10. 复盘：这轮学习中发现的新经验（新的坑、更好的 prompt 写法）**回写进本 skill**——工作流靠迭代变值钱。
