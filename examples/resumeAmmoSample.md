# 样例：面试要点速查（根目录 面试要点.md）的写法范式

> **来自**：一个 FastAPI + LangChain + pgvector 的 AI 面试项目的真实产物节选（亮点表 + STAR 故事 + 追问清单）。
> **对应产物**：根目录 `面试要点.md`（生成规范见 references/endToEndWorkflow.md §3.7）。
> **仿写时替换**：亮点内容、公式、数字换成目标项目的真实实现；**保留的是双栏范式**——每个亮点 = 一句话讲法（张口就来）+ 深挖支撑（被追问时续命），追问清单 = 简答一句 + 指路到细节文档。

---

## 节选一：项目亮点表（一句话讲法 + 深挖支撑 双栏）

| # | 亮点 | 一句话讲法 | 被深挖时的支撑点 |
|---|------|-----------|-----------------|
| 1 | RAG 三级降级出题 | "按题库召回量走选题/补全/纯生成三分支，出题永不失败" | 召回层还有一级：放宽 position_tag 二次召回；阈值 0.7 宁缺毋滥；source 标记可追溯 |
| 2 | 采分点锚定评分 | "评分注入题库参考答案与采分点，稳定可解释" | 题库题从召回起就携带 key_points 直到评分 prompt；纯 AI 题没有采分点→质量分层的依据 |
| 3 | Agent 岗位匹配 | "tool-calling 串 5 个工具，LLM 干判断、规则干计算" | 打分公式 0.6/0.15/0.1/0.15；max_iterations=10；中间步骤摘要可视化 |
| 4 | SSE 流式评分 | "评语流式首字 ~1s，分数殿后正则提取，体验与结构化兼得" | 前端必须 fetch+ReadableStream；X-Accel-Buffering:no；渲染时过滤 JSON 防闪现 |
| 5 | 结构化输出兜底 | "_extract_json 三层降级，永不裸信 LLM 输出格式" | 直解→抠 ```json``` 围栏→raw_decode；全项目统一复用（含 Agent 输出） |

## 节选二：STAR 故事（技术设计类）

**故事：出题链路的可用性设计**
- S：题库不可能覆盖所有岗位，但面试流程不能断，质量还要有保障
- T：优先用人工审核的题库题（带采分点），不足时无感降级
- A：pgvector 语义召回（0.7 阈值、2 倍召回量）→ 放宽岗位标签二次召回 → 按数量三分支（AI 选题/种子补全/纯生成），每题标记来源
- R：任何岗位出题零失败；题库命中时评分有采分点锚定，稳定性显著优于自由打分

## 节选三：高频追问清单（简答 + 指路）

**RAG**（详见 项目RAG实现原理.md）
1. 完整链路讲一遍 → 灌数据/向量化/存储/召回/生成五段
2. 为什么 pgvector？→ 量级+运维；说得出何时迁独立向量库
3. min_score 怎么定的？→ 0.7；调高召回空、调低进脏题；用检索测试看分布

**灵魂拷问**
4. 项目最大短板？→ **诚实**："测试基本是占位，我后来给核心函数补了单测（讲二开）；再往后想补 LLM 调用的可观测与成本统计"
5. 这是你从零写的吗？→ 诚实路径："我深入研究了一个开源项目，吃透其架构，反推了 PRD 和技术规格，并做了二次开发"——fork 关系一查即穿帮，而"读懂复杂项目+能二开"本身就是公司要的能力

## 这个样例展示的范式要点（仿写自查）

1. 亮点表严格双栏：左边一句话是**电梯演讲**，右边支撑点是**追问弹药**——两栏内容不重复。
2. STAR 的 A 里有具体机制和数字，R 有可验证的效果，不写"提升了系统稳定性"这种空话。
3. 追问清单每条控制在一行：简答给骨架，细节**指路**到深潜文档，本文档自身保持"30 分钟能扫完"的体量。
4. 短板题和"从零写的吗"必须预置**诚实答案**——这两题答虚了，前面全白讲。
